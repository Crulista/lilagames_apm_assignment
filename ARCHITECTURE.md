# Architecture

## System Overview

The Player Journey Visualizing Tool is a two-service system: a Python backend that processes raw telemetry data, and a React frontend that renders the visualization. This separation exists because the .nakama-0 parquet files contain binary-encoded event columns that only Python's pyarrow can reliably decode. JavaScript parquet readers (hyparquet, duckdb-wasm) were tested and could not handle the bytes columns.

## Tech Stack

| Layer | Choice | Why |
|-------|--------|-----|
| Backend | Python 3.12, FastAPI, PyArrow, Pandas | Only reliable way to read .nakama-0 parquet files with binary event columns |
| Frontend | React 18, Vite | Fast development, component model fits the interactive UI |
| Rendering | HTML5 Canvas 2D | Direct pixel control for paths, emoji markers, heatmap compositing, and compass overlay |
| Heatmaps | Custom adaptive radial gradient renderer | 55 lines, auto-tunes radius and sampling based on point density |
| Insights | Code-based analysis engine (no API calls) | Extraction-shooter-specific pattern detection, runs instantly on any data |
| Upload Progress | Server-Sent Events (SSE) | Simpler than WebSockets, one-directional, works through proxies |
| Frontend Hosting | Vercel | Free, zero-config static deployment |
| Backend Hosting | Railway | Free trial, supports Python with pyarrow, handles long-running zip processing |

## Coordinate Mapping

The README provides world-to-minimap conversion parameters per map:

| Map | Scale | Origin X | Origin Z |
|-----|-------|----------|----------|
| AmbroseValley | 900 | -370 | -473 |
| GrandRift | 581 | -290 | -290 |
| Lockdown | 1000 | -500 | -500 |

Conversion formula:
```
u = (world_x - origin_x) / scale       -> UV horizontal (0-1)
v = (world_z - origin_z) / scale       -> UV vertical (0-1)
pixel_x = u * 1024                      -> Minimap X
pixel_y = (1 - v) * 1024               -> Minimap Y (flipped for image coords)
```

The Y column in the data is elevation (3D height), NOT a map coordinate. Only X and Z are used. The Y-flip on v accounts for image coordinates having origin at top-left vs the game's bottom-left.

All pixel coordinates are normalized to a 0-1024 space in preprocessing, then scaled to actual canvas size at render time using `scale = canvasSize / 1024`. This keeps data portable regardless of display resolution.

Map config is stored in `meta.json` and loaded dynamically. Unknown maps use a fallback config. New maps added via uploaded data will appear in dropdowns automatically.

## Timestamp Handling

Match timestamps span sub-second ranges (typically 300-900ms across all players). This is the actual telemetry recording window. The timeline shows percentage progress (0-100%) rather than clock time, with configurable playback speeds from 0.1x to 5x where 1x plays through the full recording in 5 seconds.

Timestamps are stored as nanoseconds (pandas `.value`) for Windows compatibility. The `.timestamp()` method was avoided because it can return incorrect values for dates near epoch on Windows.

## Adaptive Heatmap Rendering

The heatmap renderer auto-adjusts based on how many data points exist for the selected overlay:

| Point Count | Radius | Sampling | Intensity | Use Case |
|-------------|--------|----------|-----------|----------|
| Under 50 | 24px | All points | 4x (glow) | Storm deaths, rare events |
| 50-500 | 16px | All points | 2.5x | Kills, deaths |
| 500-2000 | 12px | All points | 2x | Loot pickups |
| 2000+ | 6px | Every Nth point | 1.8x | Traffic/movement |

This prevents the "everything is yellow" problem with traffic overlays while ensuring rare events like storm deaths are clearly visible with a glow effect.

A global opacity slider in the Analytics tab lets the designer see the map underneath the overlay.

## Insights Engine

The insights engine (192 lines, no API calls) generates level-designer-actionable analysis by reading the current match data or heatmap data. It runs extraction-shooter-specific checks:

For individual matches:
- Bot ratio analysis (is the match dominated by bots?)
- Bot farming pattern detection (are players hunting bots instead of engaging PvP?)
- Storm casualty analysis with quadrant identification
- Combat hotspot detection (is fighting concentrated in one area?)
- Loot concentration analysis
- Map coverage percentage (what % of the map are players actually using?)
- Early vs late game combat balance

For heatmap overlays:
- Spatial distribution by quadrant (NW/NE/SW/SE)
- Imbalance detection with type-specific suggestions
- Underserved zone identification
- Day-over-day variance analysis

Each insight includes a title, explanation, metric, and severity level (warning/info/ok).

## Upload Flow

1. Designer selects a zip file in the Upload tab
2. Frontend sends POST to `/api/upload` with the file
3. Backend returns a `job_id`
4. Frontend opens SSE connection to `/api/progress/{job_id}`
5. Backend processes in a background thread, streaming updates:
   - "Extracting zip..."
   - "Reading files: 342/1243 (2 failed)"
   - "Building heatmaps..."
   - "Done! 796 matches from 1243 files, 3 maps, 5 days."
6. On completion, frontend fetches processed data from backend API
7. Data loads into the existing visualization (same as preprocessed data)
8. Minimap images are extracted from the zip and served via `/api/data/{job}/minimap/{map_name}`

If processing fails, specific error messages explain why:
- "Invalid zip file"
- "No .nakama-0 or .parquet files found"
- "Failed to read [filename]: [error]"

## Data Flow: Two Paths

**Path 1: Preprocessed (local development)**
```
Raw .nakama-0 files -> process_data.py -> JSON in public/data/ -> Frontend reads static files
```

**Path 2: Uploaded (production use)**
```
Zip upload -> Backend API -> Processed in memory -> Frontend fetches from API endpoints
```

Both paths produce the same data structures. The frontend does not know or care which path was used.

## Trade-offs

| Decision | Alternative Considered | Why This Choice |
|----------|----------------------|-----------------|
| Two-service architecture | Single static site | .nakama-0 binary columns cannot be read by any JavaScript parquet library. Python is required. |
| Canvas 2D | deck.gl / WebGL | 89K events is well within Canvas 2D performance limits. WebGL adds complexity without benefit at this scale. |
| Per-match JSON files | Single monolithic file | Lazy loading keeps initial page load under 1 second. Each match file is under 50KB. |
| Custom heatmap renderer | heatmap.js library | 55 lines, renders on the same canvas, no dependency. Adaptive density tuning would be harder with a library. |
| SSE for progress | WebSockets | One-directional updates only. SSE is simpler, works through CDN proxies, auto-reconnects. |
| Code-based insights | LLM API calls | Zero cost, zero latency, works offline, never fails. Can mention "LLM-powered insights" as a future enhancement. |
| Percentage timeline | Clock-time display | Timestamps span sub-second ranges. Showing "0:00 / 0:00" is misleading. Percentage is honest and still useful. |
